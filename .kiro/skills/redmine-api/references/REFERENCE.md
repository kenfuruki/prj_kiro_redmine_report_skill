## ページネーション

大量データ取得時のパラメータ：
- `limit`: 1回の取得件数（デフォルト25、最大100）
- `offset`: 開始位置
- `total_count`: レスポンスに含まれる総件数

例: 26件目から50件取得
```bash
curl -s -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  "$REDMINE_URL/issues.json?limit=25&offset=25"
```

## フィルタパラメータ

チケット一覧で使用可能なフィルタ：

| パラメータ | 説明 | 例 |
|-----------|------|-----|
| status_id | ステータス（open/closed/*） | `status_id=open` |
| assigned_to_id | 担当者ID（meで自分） | `assigned_to_id=me` |
| tracker_id | トラッカーID | `tracker_id=1` |
| priority_id | 優先度ID | `priority_id=4` |
| created_on | 作成日（>=, <=, ><で範囲指定） | `created_on=>=2025-01-01` |
| updated_on | 更新日 | `updated_on=>=2025-04-01` |
| due_date | 期日 | `due_date=<=2025-05-01` |
| sort | ソート（フィールド:asc/desc） | `sort=updated_on:desc` |

## エラーレスポンス対処

| HTTPステータス | 意味 | 対処 |
|---------------|------|------|
| 401 Unauthorized | APIキーが無効 | APIキーを再確認する |
| 403 Forbidden | 権限不足 | 対象リソースへのアクセス権を確認する |
| 404 Not Found | リソースが存在しない | ID・URLを確認する |
| 422 Unprocessable Entity | バリデーションエラー | 必須フィールドや値の形式を確認する |
| 500 Internal Server Error | サーバーエラー | 時間を置いて再試行する |

## ダッシュボードレポート

### CSVフォーマット

```csv
project_identifier,redmine_identifier,ai_flag
GJ25500801,gj25500801_-2026,1
AH26501601,,1
```

| カラム | 必須 | 説明 |
|--------|------|------|
| project_identifier | はい | rpm.csvの子案件No |
| redmine_identifier | いいえ | Redmine識別子（直接指定したい場合）。空欄の場合はカスタムフィールドから自動解決 |
| ai_flag | いいえ | AI考察を有効にする場合: `1` / `true` / `yes` / `○` |

### 制限事項

| 項目 | 値 |
|------|-----|
| CSVプロジェクト上限 | 30件 |
| APIリクエスト間隔 | 0.5秒（毎リクエスト後） |
| AI考察の上限 | 10件（ai_flag=True + 期限リスク検知の合計、CSVの記載順で優先） |
| 定点数 | 4（3週間前・2週間前・1週間前・現在） |

### 状態復元の仕組み

ダッシュボードの4定点データは、チケットの `journals`（変更履歴）から過去の状態を復元して算出しています。

- ステータス: `journals[].details` の `status_id` 変更を逆順に巻き戻し
- 進捗率: `journals[].details` の `done_ratio` 変更を逆順に巻き戻し
- 対象定点より後の変更を巻き戻すことで、その時点の状態を再現

### サブプロジェクトの扱い

- `/projects/{id}.json?include=children` でサブプロジェクトを検出
- `fetch_all_subproject_ids()` で再帰的に全階層のサブプロジェクトを取得
- `_visited` セットで循環参照を防止
- サブプロジェクトのチケットは親プロジェクトに合算
- ダッシュボード上は親プロジェクト単位で表示

### 識別子解決の仕組み

`project_identifier`（子案件番号）はRedmineのidentifier（識別子）とは異なるため、起動時に以下のマッピングを構築する。

1. `/projects.json` を全件ページネーション取得
2. 各プロジェクトのカスタムフィールド id=233（子案件番号）の値を取得
3. `{子案件番号: identifier}` のマップを構築
4. 同じ子案件番号が複数ある場合は `parent` フィールドなし（トップレベル）を優先

解決の優先順位:
1. CSVの `redmine_identifier` 列に値がある → そのまま使用
2. マップに子案件番号がある → マップから解決
3. どちらもない → スキップ（stderrに警告）

### 進捗報告トラッカーの取得ロジック

1. プロジェクト内の「進捗報告」トラッカーのチケットを全件取得
2. `updated_on` が最新のチケットを選択
3. そのチケットの最新コメント（`journals` の `notes`）があればそれを表示
4. コメントがなければチケットの `description`（説明文）を表示
5. 表示は先頭300文字で切り詰め、「続きを読む」で全文展開

### AI考察のdata属性

HTMLのAI考察プレースホルダーには以下のdata属性が埋め込まれます：

| 属性 | 内容 |
|------|------|
| `data-project` | プロジェクトidentifier |
| `data-project-name` | プロジェクト名 |
| `data-recent-topics` | 直近1週間の課題チケットタイトル（セミコロン区切り） |
| `data-summary` | トラッカー別の現在件数と解決・終了数 |
| `data-progress-report` | 進捗報告の最新テキスト（先頭500文字） |
| `data-risk-scores` | 4定点のRisk_Score推移（例: `3,5,4,7`）。Deadline_Risk_Flag=Trueのプロジェクトのみ |

### 期限リスク検知

ダッシュボードは、課題チケットとQ/Aチケットの期限管理状況を4定点で分析し、上昇傾向にあるプロジェクトを自動検知します。

#### Risk_Score算出ロジック

各プロジェクト・各Checkpoint（4定点）について、以下の合計値をRisk_Scoreとして算出します。

```
Risk_Score = Overdue_Count + Blank_Due_Count
```

| 指標 | 定義 |
|------|------|
| Overdue_Count | `due_date` がCheckpoint日より前であり、かつステータスが未完了のチケット数 |
| Blank_Due_Count | `due_date` が未設定（空欄）であり、かつステータスが未完了のチケット数 |

- **対象トラッカー:** 課題、Q/A のみ（サポート・成果物・進捗報告は対象外）
- **未完了の定義:** ステータスが「解決」「終了」「却下」以外

#### 完了ステータスの除外条件（CLOSED_STATUSES）

以下のステータスのチケットは、`due_date` の値に関わらずRisk_Scoreの算出対象から除外されます。

| ステータス名 | 説明 |
|-------------|------|
| 解決 | 対応完了（レビュー待ち等） |
| 終了 | 完全に完了 |
| 却下 | 対応不要と判断 |

```python
CLOSED_STATUSES = {"解決", "終了", "却下"}
```

#### 上昇傾向の判定基準（Trend_Analysis）

4定点のRisk_Score推移から、以下の条件で上昇傾向を判定します。

| 条件 | 判定結果 |
|------|---------|
| 3区間（3w→2w, 2w→1w, 1w→now）のうち2区間以上でRisk_Scoreが増加 | `Deadline_Risk_Flag = True`（上昇傾向） |
| 4定点すべてRisk_Score = 0 | `Deadline_Risk_Flag = False` |
| 上記以外 | `Deadline_Risk_Flag = False` |

判定例:
```
[3, 5, 4, 7] → 3→5（増加）, 5→4（減少）, 4→7（増加） → 2区間増加 → True
[0, 0, 0, 0] → 全て0 → False
[5, 3, 2, 1] → 全区間減少 → False
[1, 2, 3, 2] → 1→2（増加）, 2→3（増加）, 3→2（減少） → 2区間増加 → True
```

#### Deadline_Risk_Flagの表示

| 表示場所 | 内容 |
|---------|------|
| 一覧テーブル「期限リスク」列 | `⚠ 期限リスク` バッジを表示 |
| 一覧テーブル ツールチップ | 4定点のRisk_Score推移（例: 「3→5→4→7」） |
| 詳細ページ | Risk_Score推移テーブル（4定点の数値） |
| AI考察 | `data-risk-scores` 属性にRisk_Score推移を埋め込み |

### バグフェーズ分類

バグトラッカーのチケットは、`fixed_version`（バージョン）フィールドの名前に含まれるフェーズコードで分類されます。

| フェーズコード | 色 | 説明 |
|--------------|-----|------|
| 05_PS | 水色 | 製造・単体テスト |
| 06_IT | 青 | 結合テスト |
| 07_ST | オレンジ | システムテスト |
| 08_UAT | 緑 | ユーザー受入テスト |
| 09_OT | グレー | 運用テスト・移行 |
| その他 | 薄グレー | 上記に該当しないバージョン |

分類ロジック: バージョン名がフェーズコードで始まるか、フェーズコードを含む場合にそのフェーズに分類。

### 工程別 成果物・テスト進捗テーブル

プロジェクト詳細ページのトップに表示される、バージョン（工程）ごとの進捗サマリーテーブル。

#### 対象工程

| 工程 | 種別 | 集計対象 |
|------|------|---------|
| SA（02_SA） | 成果物 | 成果物トラッカーのチケット |
| UC（03_UC） | 成果物 | 成果物トラッカーのチケット |
| SS（04_SS） | 成果物 | 成果物トラッカーのチケット |
| IT（06_IT） | テストケース | テストスイートトラッカーのカスタムフィールド |
| ST（07_ST） | テストケース | テストスイートトラッカーのカスタムフィールド |
| UAT（08_UAT） | テストケース | テストスイートトラッカーのカスタムフィールド |

SP、PG、PT、OT工程は除外。取消・却下ステータスのチケットも除外。

#### 成果物（上流〜設計）の集計ロジック

- **総数**: 成果物トラッカーのチケット数（取消除外）
- **予定数**: 期日（`due_date`）が現時点で到来済みのチケット数
- **完了数**: ステータスが「終了」のチケット数
- **達成率**: 完了数 ÷ 予定数 × 100%

#### テストケース（テストフェーズ）の集計ロジック

- **総数**: テストスイートトラッカーのチケット数（スイート数）
- **予定ケース数**: カスタムフィールド「予定ケース数」の合計
- **実績ケース数**: カスタムフィールド「実績ケース数」の合計
- **達成率**: 実績ケース数 ÷ 予定ケース数 × 100%

> カスタムフィールド名は仮設定（`予定ケース数`、`実績ケース数`）です。実環境に合わせて `dashboard_report.py` の `CF_PLANNED_CASES`、`CF_ACTUAL_CASES` を変更してください。
