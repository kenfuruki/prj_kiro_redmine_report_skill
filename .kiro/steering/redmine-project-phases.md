# プロジェクトフェーズ定義

Redmineで管理するプロジェクトは、以下のフェーズで進行する。
AI考察やダッシュボード分析時に、プロジェクトの現在フェーズを考慮すること。

## フェーズ一覧

| コード | フェーズ名 | 説明 |
|--------|-----------|------|
| 00_共通 | 共通 | プロジェクト横断の共通タスク・管理業務 |
| 01_SP | システム企画 | システム企画・構想フェーズ |
| 02_SA | システム分析 | 要件分析・業務分析フェーズ |
| 03_UC | ユースケース | ユースケース定義・要件定義フェーズ |
| 04_SS | システム設計 | 基本設計・詳細設計フェーズ |
| 05_PS/PG/PT | 製造・単体テスト | プログラム設計・製造・単体テストフェーズ |
| 06_IT | 結合テスト | 結合テスト（Integration Test）フェーズ |
| 07_ST | システムテスト | システムテスト（System Test）フェーズ |
| 08_UAT | ユーザー受入テスト | ユーザー受入テスト（User Acceptance Test）フェーズ |
| 09_OT | 運用テスト・移行 | 運用テスト・本番移行フェーズ |

## フェーズ別の注目ポイント

AI考察を生成する際は、プロジェクトの現在フェーズに応じて以下の観点を重視すること。

### 上流フェーズ（01_SP 〜 03_UC）
- 課題・Q/Aの件数が多いのは正常（要件の洗い出し段階）
- 成果物の進捗率よりも、課題の解決率に注目
- 未解決の課題が滞留していないか

### 設計フェーズ（04_SS）
- 成果物（設計書）の予実が重要
- 課題の新規発生が減少傾向にあるか
- Q/Aが解決に向かっているか

### 製造・テストフェーズ（05_PS/PG/PT 〜 07_ST）
- 課題の急増は要注意（バグの多発を示唆）
- 成果物の遅延はテスト全体に波及するリスク
- 解決・終了の消化ペースが新規発生を上回っているか

### 受入・移行フェーズ（08_UAT 〜 09_OT）
- 未解決の課題が残っていないか（リリースブロッカー）
- 成果物がすべて完了に近づいているか
- サポートチケットの増加は運用準備の不足を示唆
